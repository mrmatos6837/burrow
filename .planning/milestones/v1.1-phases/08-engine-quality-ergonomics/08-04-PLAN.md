    - find command produces same output as before but uses tree.findById/getPath internally
    - PREVIEW_TRUNCATE_LENGTH constant exists in mongoose.cjs (was inline 80)
    - renderTree with non-numeric depth throws or returns error
  </behavior>
  <action>
1. **CLI input validation (VALID-01, VALID-02, VALID-03):**

   **--depth validation (VALID-01):** In the read command, after `parseInt(values.depth, 10)`, add:
   ```javascript
   if (values.depth !== undefined && isNaN(depth)) {
     handleError('--depth must be a number');
   }
   ```

   **--at validation (VALID-02):** In add and move commands, after `parseInt(values.at, 10)`, add:
   ```javascript
   if (values.at !== undefined) {
     if (isNaN(position)) handleError('--at must be a number');
     if (position < 0) handleError('--at must be non-negative');
   }
   ```

   **Unknown flags (VALID-03):** Change `strict: false` to `strict: true` on ALL parseArgs calls (add, edit, remove, read, dump, archive, unarchive, path, find). This means every allowed option MUST be declared in the options object. Most commands already declare --width. Ensure all legitimate flags are declared. Note: move already uses `strict: true`.

2. **renderTree depth validation (QUAL-03):** In mongoose.cjs renderTree, add a type check at the top:
   ```javascript
   if (depthArg !== undefined && typeof depthArg !== 'number') {
     throw new Error('renderTree: depth must be a number');
   }
   ```
   This catches cases where a string accidentally leaks through.

3. **find command rewrite (API-02):** Replace the inline `searchCards` recursive function with tree engine usage. Use `findById` + `getPath` pattern: walk all cards once using a generic tree walker (or use the existing recursive pattern but call `tree.getPath(data, card.id)` for matched cards to get the path). Since getPath already exists and returns the ancestry, the find command should:
   - Walk all cards to find matches (keep the recursive walk for matching — it's necessary)
   - For each match, use `tree.getPath(data, card.id)` to get the path instead of manually accumulating `[...ancestors, card]` with O(n) copies
   - Actually, the current approach of passing ancestors down is MORE efficient than calling getPath per match (getPath does a full tree walk per call). So the better fix is: keep the recursive search but use the tree engine's `findById` to confirm and extract the ancestor array only for matches. OR simply acknowledge that the current approach is reasonable and the "reuse tree engine" means extracting the search into mongoose.cjs as a proper `findCards(data, query)` function.
   - Best approach: Add a `searchCards(data, query)` function to mongoose.cjs that encapsulates the recursive search, returning `[{id, title, path: [{id, title}...]}]`. The CLI's find command calls this instead of reimplementing recursion inline.

4. **Magic number constants (QUAL-04):** In mongoose.cjs, rename the inline `80` in makePreview to a named constant `PREVIEW_TRUNCATE_LENGTH = 80` at the top of the file. In render.cjs, BODY_TRUNCATE_LENGTH (200) and DIFF_TRUNCATE_LENGTH (40) are already named constants. Add a JSDoc comment to each explaining what they control. The MIN_TERM_WIDTH (40) already has a derivation comment.

5. **Add CLI tests:** Test --depth with non-numeric value. Test --at with negative value. Test unknown flag rejection on at least 2 commands (read, add).
